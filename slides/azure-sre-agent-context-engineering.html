Azure SRE Agent 的经验：为什么 100+ 工具和 50+ 专家代理，最后不如 5 个宽工具

这篇是少见的生产经验复盘：微软在构建 Azure SRE Agent 时发现，复杂系统的很多问题并不是靠加更多专家代理和更多工具解决，反而常常要靠删减、收敛和外化状态来解决。

第 1 页

最反直觉的结论

100+ 工具和 50+ 代理并没有让系统更稳，最后反而是 5 个核心宽工具更可靠。

少量宽工具往往比大量窄工具更适合 agent。
多代理 handoff 有很高的语义损耗成本。
第 2 页

多代理交接有成本

每一次 handoff 都会引入状态损失、语义变形和重复探索。

少量宽工具往往比大量窄工具更适合 agent。
多代理 handoff 有很高的语义损耗成本。
第 3 页

把知识和状态迁出 prompt

文件、skills、会话工件和代码解释器，比长 prompt 更适合承载复杂任务。

多代理 handoff 有很高的语义损耗成本。
LLM 更适合编排和解释，不适合承担大量机械计算。
第 4 页

LLM 应该做什么

它更像 orchestrator 和解释器，而不是直接做重计算的执行器。

LLM 更适合编排和解释，不适合承担大量机械计算。
知识与状态应该尽量外化到文件、skills 和数据工件。
第 5 页

我的结论

生产级 agent 工程常常靠删减复杂度来赢得可靠性。

知识与状态应该尽量外化到文件、skills 和数据工件。
生产可靠性常常来自减法，而不是加法。
