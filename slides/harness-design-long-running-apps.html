Harness design for long-running application development：让 AI 在数小时任务中保持节奏的三代理架构

Anthropic Labs 团队的 Prithvi Rajasekaran 在两个领域（主观审美与客观正确性）各建一套 harness，最终都收敛到同一个 GAN 启发的三代理架构：Planner 规划、Generator 构建、Evaluator 评估。

第 1 页

朴素实现的两类失败

长任务有两个持续存在的失败模式：上下文焦虑（context anxiety）和自我评估失真。

Context reset 可以解决上下文焦虑，但增加编排复杂度和 token 开销。
分离生成与评估是打破"自信自夸"循环最有效手段。
第 2 页

GAN 启发的设计评估体系

四个评分标准：Design quality、Originality、Craft、Functionality。有意强调前两项，让模型在创意上走出安全区。

Evaluator 使用 Playwright MCP 直接与运行中的页面交互验证。
"博物馆质量"这样的措辞直接塑造了输出特征的视觉收敛方向。
第 3 页

三代理架构

Planner（扩展 spec）、Generator（逐 sprint 构建）、Evaluator（Playwright 验证）。每个 sprint 前双方协商 contract。

Evaluator 的价值取决于任务是否超出模型可靠独立完成的能力边界。
Sprint contract 弥合了高层 spec 与可测试实现之间的差距。
第 4 页

DAW 构建实验结果

3h50m，$124，总计 3 轮 build/QA。Generator 跑了 2+ 小时无 sprint 分解。Evaluator 持续捕捉核心功能缺口。

完整 harness 输出远优于 solo agent，核心功能真正可用。
Playwright QA 发现 spec 明确要求的"核心交互"是展示态而非可操作。
第 5 页

Opus 4.6 时代的简化

移除 sprint 构造，Evaluator 移至单次 end-of-run 通过。某些原本承重的组件在更强模型下变成多余。

harness 是对模型能力边界的时刻快照，需要随模型改进持续重新评估。
Evaluator 不是固定 yes/no，而是任务复杂度的连续函数。
