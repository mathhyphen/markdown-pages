📖 Martin Fowler 权威解读

Harness Engineering
深度解析

软件架构大师 Martin Fowler 对 AI Agent 系统设计的深刻洞察
用最通俗的语言，带你理解 Guides、Sensors、Steering Loop 的精髓

🎯 核心定义：什么是 Harness？

📐 最核心的公式

Agent = Model + Harness

Harness = 除了模型本身之外的一切东西
包括：系统提示词、工具定义、检索机制、编排逻辑、规则文件……

🔄 三层同心圆模型

Martin Fowler 用三层同心圆来解释 Harness 在不同语境下的含义：

🎯

三层结构示意图

内圈（核心）：Model（模型本身，如 GPT-4）

中圈：Builder's Harness（构建者提供的Harness，如 Claude Code 自带的工具和提示词）

外圈：User's Harness（用户自定义的Harness，针对你的具体场景定制）

📌 本文重点关注：User's Harness（用户层）

✨ 好的 Outer Harness 有两大目标

🎯 目标1：提高首次成功率

通过前馈控制（Guides），让 Agent 第一次就做对，减少返工。

🔄 目标2：自动自我纠正

通过反馈控制（Sensors），让 Agent 能在人类介入前自己发现问题并修复。

最终效果：减少人工审查负担 + 提高系统质量 + 节省 Token 成本

🔑 核心概念：Guides vs Sensors

这是整篇文章最重要的两个概念！理解了这两个，你就理解了 Harness Engineering 的精髓。

➡️
Guides（引导/前馈控制）

📍 定义

在 Agent 行动之前进行干预，
预判可能的问题并提前引导。

⏰ 时间点

Feedforward（前馈）

Agent 还没开始做事的时候就已经发挥作用。

🎯 核心作用

提高首次做对的概率

就像给马套上缰绳，让它一开始就走对方向。

💡 生活类比

GPS 导航

在你出发前就告诉你："走这条路，不要走那条路"。这就是前馈控制——提前引导你避免错误路线。

⬅️
Sensors（传感器/反馈控制）

📍 定义

在 Agent 行动之后进行检测，
观察结果并提供纠正信号。

⏰ 时间点

Feedback（反馈）

Agent 完成动作后才启动检测。

🎯 核心作用

让 Agent 自我发现错误并修正

就像汽车的安全气囊，出问题时自动触发保护。

💡 生活类比

拼写检查器

你打完字后它才检查，发现错误时用红色波浪线提醒你修改。这就是反馈控制——事后检测并提示纠正。

⚠️ 为什么两者缺一不可？

情况 结果 问题
只有 Sensors（只有反馈） Agent 会不断重复同样的错误 ❌ 浪费时间和Token
只有 Guides（只有前馈） Agent 遵守规则但不知道效果如何 ❌ 无法学习和改进
Guides + Sensors（两者兼备） Agent 既被正确引导，又能从错误中学习 ✅ 最佳实践！

⚙️ 两种执行类型：Computational vs Inferential

Guides 和 Sensors 都可以以两种方式实现：计算型（确定性）或推断型（AI驱动）。

Computational 计算型

⚡ 特点

  • 确定性：相同输入永远产生相同输出
  • 快速：毫秒到秒级完成
  • 便宜：CPU 运行，成本低
  • 可靠：结果可预测、可复现

🛠️ 典型示例

Guides 示例：

  • LSP（语言服务器协议）提供代码补全
  • 代码格式化工具（Prettier、Black）
  • 项目脚手架生成脚本
  • .cursorrules / AGENTS.md 规则文件

Sensors 示例：

  • ESLint / Pylint 静态分析
  • TypeScript 类型检查器
  • 单元测试（pytest、Jest）
  • 架构约束检查（ArchUnit）
Inferential 推断型

🧠 特点

  • 语义理解：能理解代码含义和意图
  • 较慢：秒到分钟级完成
  • 较贵：GPU/NPU 运行，成本较高
  • 概率性：结果有一定随机性

🛠️ 典型示例

Guides 示例：

  • AGENTS.md 中描述性的编码规范
  • Skill 文件中的自然语言指令
  • How-to 文档和最佳实践指南
  • API 文档的自然语言描述

Sensors 示例：

  • AI Code Reviewer（LLM-as-Judge）
  • 语义重复代码检测
  • 代码质量评分（基于语义理解）
  • 安全漏洞智能扫描
💡 如何选择？
场景 推荐类型 原因
每次提交都要跑的检查 Computational 快速、便宜、可靠
需要语义理解的复杂判断 Inferential 能处理模糊性和上下文
关键路径上的核心质量门禁 两者结合 先用计算型快速筛选，再用推断型深度检查

💡 具体实例：实际项目中怎么用？

📋 Martin Fowler 提供的真实案例表

让我们看看在实际编码 Agent 场景中，Guides 和 Sensors 分别长什么样：

功能 方向 类型 具体实现示例
编码规范 Guide Inferential AGENTS.md 文件、Skill 定义文件
新项目初始化指引 Guide Both 包含指令的 Skill + Bootstrap 脚本
自动代码重构 Guide Computational 使用 OpenRewrite recipes 的工具
架构约束检查 Sensor Computational pre-commit hook 运行的 ArchUnit 测试
代码审查指导 Sensor Inferential Code Review Skill（告诉 Agent 怎么审查）

🔍 让我们深入看几个具体例子

📝 案例1：AGENTS.md 作为 Inferential Guide

这是什么？

一个放在项目根目录的 Markdown 文件，用自然语言告诉编码 Agent 这个项目的规则和约定。

示例内容：

# AGENTS.md

## 项目概述
这是一个使用 FastAPI + React 的全栈 Web 应用。

## 编码规范
- 使用 TypeScript 严格模式
- 所有 API 端点必须有输入验证
- 组件必须使用函数式写法 + Hooks
- 测试覆盖率要求 > 80%

## 架构约束
- 前端调用后端只能通过 /api/* 路径
- 数据库访问必须通过 Repository 层
- 禁止在前端直接操作 DOM（使用 React 方式）

## Git 工作流
- 功能分支命名：feature/JIRA-ID-description
- PR 必须关联 JIRA ticket
- main 分支受保护，不能直接 push

为什么这是 Guide？

✅ 前馈：Agent 开始工作前就读取这些规则
✅ Inferential：使用自然语言，需要 Agent 语义理解
✅ 作用：提高首次写出符合规范代码的概率

🔍 案例2：ESLint 作为 Computational Sensor

这是什么？

一个静态代码分析工具，能在 Agent 写完代码后立即检测语法错误、风格违规等问题。

工作流程：

# Agent 写完代码后...

# Sensor 自动运行：
$ npx eslint src/new-feature.tsx

# 输出示例：
#   ✗  12:3   error  Missing semicolon
#   ✗  25:10  warning  Unused variable 'temp'
#   ✗  48:7   error  React Hook 'useEffect' is missing dependencies

# Agent 收到这些信号后自动修复...

为什么这是 Sensor？

✅ 反馈：Agent 完成代码后才运行检测
✅ Computational：确定性的规则匹配，毫秒级完成
✅ 作用：让 Agent 自动发现并修复基础错误

🤖 案例3：AI Code Reviewer 作为 Inferential Sensor

这是什么？

使用 LLM（如 GPT-4）作为 Judge，对 Agent 产出的代码进行深度的语义级审查。

它能发现什么？

✅ 能发现的
  • 语义重复的代码逻辑
  • 过度工程化的设计
  • 潜在的边界条件 bug
  • 不符合业务意图的实现
❌ 不能完全依赖的
  • 需求理解是否正确（这需要人判断）
  • 是否过度设计（主观性强）
  • 性能瓶颈（需要实测）

关键洞察：

Inferential Sensor 特别强大的一点：它可以输出专门为 LLM 优化的纠错指令！
这就像是一种"正向的 Prompt Injection"——不是攻击，而是帮助 Agent 更好地自我修正。

🔄 Steering Loop（转向循环）

👨‍💼 人类的角色：不是骑手，而是驯马师

Martin Fowler 强调：人类的工作是"steer"（驾驶/引导）Agent，而不是每次都亲自去做。

🔄 Steering Loop 的完整流程

①

观察问题

发现 Agent 反复犯同类错误

②

改进 Harness

添加新的 Guide 或加强 Sensor

③

验证效果

观察改进后是否解决问题

④

循环迭代

持续优化直到满意

💪 用 AI 来改进 Harness

Martin Fowler 指出一个很酷的点：我们可以用 Agent 本身来帮我们构建更好的 Harness！

🤖 Agent 能帮忙做什么？

  • 编写结构性测试
  • 从观察到的模式中生成规则草案
  • 搭建自定义 Linter 的框架
  • 通过代码考古学创建 How-to 文档

🎯 这意味着什么？

  • 构建 Harness 的成本大幅降低
  • 可以更频繁地迭代优化
  • 形成正向循环：更好的 Harness → 更好的 Agent → 更好的 Harness
🎯 Steering Loop 的核心理念

不要每次都手动修复 Agent 的错误，而是改进系统让这类错误不再发生。

这就是从"救火队员"到"系统设计师"的思维转变。

⏱️ 时机选择：Keep Quality Left

📍 将检查点尽可能左移

Martin Fowler 借用了软件工程中 "Shift Left Testing"（测试左移）的理念：
越早发现问题，修复成本越低。

📅

开发时间线上的 Guides & Sensors 分布

【提交前】最快的检查
• LSP 实时代码补全（Guide）
• 快速 Linter 检查（Sensor）
• 基础单元测试（Sensor）

【Agent 工作时】第一轮自纠正
• Code Review Skill 引导（Guide/Sensor）
• ESLint/Pylint 自动运行（Sensor）
• 测试覆盖率检查（Sensor）

【集成后】CI/CD Pipeline
• 重新运行所有快速检查
• 更全面的 Code Review（Inferential Sensor）
• 架构合规性审查（Sensor）

【持续监控】生产环境
• 死代码检测（Sensor）
• 依赖安全扫描（Sensor）
• SLO 性能监控（Sensor）
• 日志异常检测（Inferential Sensor）

📊 两类持续监控

代码库漂移检测

那些随时间积累的质量退化：

  • 死代码累积
  • 测试覆盖率下降
  • 依赖版本过时
  • 技术债务增长

运行时反馈

生产环境的实时信号：

  • 延迟 SLO 是否达标
  • 错误率趋势
  • 可用性监控
  • AI Judge 抽样评估响应质量

📂 Harness 的分类维度

Martin Fowler 提出，可以根据 Harness 要监管什么来分类。
区分这些类别有助于我们用更精确的语言讨论不同类型的 Harness。

1️⃣ Maintainability Harness（可维护性 Harness）

这是目前最容易实现的类型，因为我们有很多现成的工具可以使用。

它能捕捉哪些 Agent 失败模式？

失败模式 能否被 Computational Sensor 捕捉 能否被 Inferential Sensor 捕捉
重复代码 ✅ 可靠 ✅ 可以
圈复杂度过高 ✅ 可靠 -
缺少测试覆盖 ✅ 可靠 -
架构漂移 ✅ 可靠 ✅ 可以
风格违规 ✅ 可靠 -
语义重复代码 ❌ 困难 ✅ 可能（但昂贵且不确定）
误诊问题 / 过度设计 ❌ 不能 ⚠️ 不够可靠

⚠️ 关键限制

没有任何 Sensor 能可靠地解决以下问题：

  • 问题的误诊断（Agent 理解错了需求）
  • 过度工程（做了不需要的功能）
  • 误解指令（如果人类一开始就没说清楚）

这些问题最终还需要人类的明确需求和清晰的沟通来解决。

2️⃣ Architecture Fitness Harness（架构适配性 Harness）

这一类关注系统的架构是否适合当前需求。

🏗️ Guides 示例

  • 架构决策记录 (ADR)
  • 技术选型文档
  • 模块边界定义
  • 数据流规范

🔍 Sensors 示例

  • 依赖方向检查
  • 分层违规检测
  • 循环依赖识别
  • 架构热力图可视化

3️⃣ Security & Compliance Harness（安全合规 Harness）

这一类确保系统满足安全和法规要求。

🛡️ Guides 示例

  • 安全编码规范 (OWASP)
  • 数据处理政策 (GDPR)
  • 认证授权流程文档
  • 敏感数据处理规则

🚨 Sensors 示例

  • SQL 注入检测
  • XSS 漏洞扫描
  • 依赖漏洞检查 (Dependabot)
  • 权限配置审计

🎯 核心要点总结

🌟 Martin Fowler 文章的五大核心洞察

1️⃣ 公式：Agent = Model + Harness

Harness 是除模型外的所有东西。我们的工作是设计和优化这个 Harness。

2️⃣ 双轮驱动：Guides（前馈）+ Sensors（反馈）

Guides 提高首次成功率，Sensors 实现自动纠错。缺一不可。

3️⃣ 双重类型：Computational（快/准/便宜）vs Inferential（慢/智/贵）

根据场景选择合适的类型，通常组合使用效果最佳。

4️⃣ 人类角色：Steering（驾驶）而非 Riding（骑行）

不要每次都手动修错，而是改进 Harness 让错误不再发生。可以用 AI 来辅助构建 Harness。

5️⃣ 时机原则：Quality Left（质量左移）

把检查点尽量前置，越早发现问题成本越低。建立持续监控闭环。

🔗 与 Meta-Harness 论文的关系

Martin Fowler 的文章提供了 Harness Engineering 的概念框架和分类体系（是什么、为什么、怎么做）。

Meta-Harness 论文则提供了自动化优化 Harness 的方法论（如何自动找到最优的 Harness 配置）。

两者互补：先理解 Fowler 的框架，再用 Meta-Harness 的方法去自动化优化。

💡 给初学者的行动建议

第1步：观察现状

在使用编码 Agent 时，记录它经常犯的错误类型。

第2步：添加 Guides

创建 AGENTS.md 或规则文件，预防已知问题。

第3步：添加 Sensors

配置 ESLint、测试、AI Reviewer 等自动检测。

第4步：迭代优化

根据新出现的问题持续改进 Harness。

Martin Fowler 的 Harness Engineering 核心在于：
用 Guides 提高首次成功率，用 Sensors 建立自动纠错闭环。
