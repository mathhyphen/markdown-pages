先给结论

Harness 不是让模型“更聪明”的修辞层，而是让系统“更可控、更可验证、更可积累”的执行层。

如果只看 prompt，你优化的是一次回答；如果看 context，你优化的是一次任务；如果看 harness， 你优化的是一整个工作系统。真正的 harness 设计，核心不是“我给模型说什么”，而是 我要让系统稳定地产生什么结果，并且如何在失败后留下足够证据去修正自己。

内层

Agent 执行任务，调用工具，写入状态，完成本轮工作。

中层

评估系统读取日志、样本、指标，对本轮结果给出 pass/fail 和诊断。

外层

Meta-Harness 只优化 harness 本身，不直接优化模型权重和最终科学结论。

一个实用判断标准是：如果某个失败会重复出现两次以上，它就不再是单次执行问题，而是 harness 问题。 这时应该回到系统层修 guides、sensors、contracts、evaluation，而不是继续补一句 prompt。

Harness 到底在解决什么

从 Martin Fowler 到 Anthropic、LangChain、OpenAI，再到 Meta-Harness，大家虽然表述不同， 但在本质上都在处理同一件事：模型很强，但它天然缺乏约束、持续状态、可靠自证和长期可积累的外部结构。

典型失败 单靠模型很难解决的原因 Harness 应该补的结构
任务跑偏 高层目标不等于可执行目标 contracts、阶段目标、done criteria
过早宣布完成 模型偏向第一个“看起来能交代”的答案 pre-completion checklist、强制验证、feature status
多轮后失去连贯性 上下文窗口不是长期记忆 文件化状态、progress log、structured artifacts
无法稳定复现 环境、seed、依赖和数据状态没有被记录 env lock、dataset hash、run manifest
看起来不错但其实没达标 自评偏乐观，主观判断难以稳定 独立 evaluator、硬阈值、外部 judge

通用构建框架

一个可持续的 harness，我建议按九层来理解。这个顺序不是“概念归类”，而是实际工程里更安全的搭建顺序。 前三层决定你有没有方向，中间三层决定 agent 能不能稳定工作，后面三层决定系统能不能长期迭代。

1

Goal Function

先定义最终要优化的目标函数。不是“把任务做完”，而是“什么结果可以被认为值得保留、值得晋级、值得继续投入算力”。

2

Failure Model

列出现实里最常见的失败模式。harness 是针对失败模式设计的，不是围绕模型能力想象设计的。

3

Contracts

把任务切成可验证单元。每个单元需要有输入、输出、done criteria、失败回退策略和下一步移交信息。

4

Environment

定义 agent 能在哪跑、用什么工具、看到哪些文件、有哪些限制。没有环境设计，就没有真正意义上的 harness。

5

State and Memory

把所有可恢复、可交接、可审计的信息放进文件系统或数据库，而不是留在对话上下文里。

6

Guides

前馈控制。提前告诉 agent 应该怎么做，尤其是那些一旦做错就代价很高的地方，比如数据边界、流程顺序、不可跨越的安全线。

7

Sensors

反馈控制。捕捉运行后的事实信号，包括 deterministic checks 和 inferential checks。只有 sensors 才能让系统形成纠错闭环。

8

Evaluation and Promotion

把评估和晋级规则独立出来。什么情况下保留这个 run，什么情况下淘汰，什么情况下允许进入默认流程，都要由这层决定。

9

Meta-Optimization

在外环持续优化 harness 本身。不是直接改研究结论，而是改 prompting、workflow、gates、contracts、logs、评估次序和工具使用方式。

Guides 与 Sensors 应该怎么分

Guides：提前约束

例如：必须先做 smoke run；必须按 patient-level split； 必须记录 commit、seed、dataset hash；必须在进入 full run 前通过 pilot gate。

Sensors：事后抓问题

例如：NaN 检测、OOM 检测、指标回退、泄漏检测、artifact rate、 以及 LLM reviewer 对日志和样本板做的结构化批注。

一个常见误区是把 guides 全写进 prompt，把 sensors 全留给人工 review。 这样系统会很脆。更好的做法是： guides 尽量结构化，sensors 尽量自动化，人工只处理真正需要判断的部分。

从 0 到 1 的构建顺序

下面这 12 步，是我认为更稳妥的 harness 搭建顺序。它故意把 prompt 放后面，而把目标、评估和状态前置。 原因很简单：没有目标和评估，prompt 优化会很快变成自我感动。

01
定义决策问题

不是“我要搭一个 agent”，而是“我希望这个系统替我做出什么类型的高质量决策或执行”。

02
冻结初始评估集

先拿一小组 benchmark、holdout cases、反例集做固定评估，不要一边搭 harness 一边改尺子。

03
列出失败模式

至少写出 8 到 12 个高频失败模式，并按代价排序。harness 应优先修高代价失败。

04
设计最小 contracts

把系统拆成可验证单元，每个单元定义输入、输出、验收标准、失败回退和移交信息。

05
设计环境和工具边界

先限制能力，再开放能力。明确 agent 读哪些目录、跑哪些命令、可否联网、何时写入持久状态。

06
先加 deterministic guides

把最不该出错的流程前置为硬要求，例如数据检查、命名规则、运行前清单和阶段 gate。

07
再加 deterministic sensors

测试、校验器、阈值检查、文件存在性检查、状态机合法性检查，优先覆盖最便宜的自动诊断。

08
落地状态工件

至少要有 spec、run manifest、progress、metrics、logs、artifacts summary。没有这些，后面无法做外环优化。

09
补 inferential judge

在确定性检查之后，再引入 reviewer、LLM-as-judge、样本诊断。它们是辅助，不应该替代硬评估。

10
先跑 baseline harness

记录一个朴素版本的起点。没有 baseline，就无法判断后续每次优化是否真的变好。

11
建立 trace registry

把每次 run 的代码、配置、日志、分数、失败原因和人工批注收集起来，作为 Meta-Harness 的训练土壤。

12
最后再开启 Meta-Harness

只有在评估集、日志、回滚和晋级规则都稳定后，才让外环开始优化 harness 本身。

一般情况的逻辑模板

如果你不是做编码 agent，也不是医学影像，而是任意一种 agentic workflow，都可以套用下面这张通用模板。 它的思路是：先把系统写成 blueprint，再让 agent 在 blueprint 里工作。

Meta-Harness 放在哪里

Meta-Harness 最容易被误解的地方是：很多人把它当成“自动 prompt 优化器”。这太窄了。 从 2026 年 3 月 30 日的 Meta-Harness 论文 和项目页来看，它真正优化的是整个 harness 代码，而不是一段 instruction string。

它应该优化的东西

prompt blocks、tool descriptions、contracts、completion check、context assembly、retry policy、evaluation order、logging strategy。

它不应该直接优化的东西

隐藏测试集标签、最终科学结论、人工专家判断标准，以及你用来判定是否“作弊”的规则本身。

它之所以有效，不只是因为“会搜索”，而是因为它能读到历史候选 harness 的 源码、分数和原始执行轨迹。这和只给一个总分完全不同。

前提 1：Harness 必须版本化

如果每次 run 的 prompt、middleware、规则和评估顺序都没法 diff，外环就没有真正的优化对象。

前提 2：Eval 必须冻结

Meta-Harness 只能在固定尺子上搜索；如果尺子一起变，它优化出来的是制度漏洞，不是系统质量。

前提 3：Trace 必须丰富

至少保存配置、日志、错误、样本、分数、人工批注。只有分数，没有上下文，就做不出像样的因果诊断。

前提 4：Promotion 必须走 holdout

先在 dev set 试，再在 holdout 验。否则外环会很快把 harness 优化成 benchmark specialist。

一个实用原则是：Meta-Harness 应该先优化“系统如何尝试”，再优化“系统尝试什么”。 换句话说，先改 workflow 和 gates，再去改更高风险的策略本体。

医学影像生成科研实例

如果你做的是医学影像生成科研，harness 的目标不应该是“让 agent 自动跑训练”，而应该是： 让系统稳定地产生值得你投入注意力去看的实验结果。 这里最重要的不是自动化本身，而是可复现、可审计、可比较和可晋级。

建议的角色划分

角色 职责 主要工件
Planner 把研究问题展开成实验矩阵和比较基线 spec.md, ablation_plan.yaml
Data Gatekeeper 做数据合法性、split、泄漏和预处理检查 dataset_report.json
Runner 提交训练、采样、评估任务 run_config.yaml, run_state.json
Monitor 盯训练中间状态，处理 OOM、NaN、死锁、坏 checkpoint alerts.json, runtime_log.txt
Evaluator 生成定量指标、样本板、下游任务比较 metrics.json, sample_sheet.png
Reviewer 对比 baseline，总结值得保留与不值得保留的方向 review.md

五道 gate

Gate 目的 通过条件示例
Data Gate 确保数据可信、划分合法 patient-level split、去标识完成、预处理一致、dataset card 完整
Smoke Gate 确保流程没坏 小样本可跑通、无 NaN/OOM、基础 sample sheet 正常
Pilot Gate 低成本验证方向 在 10% 到 20% 数据上不劣于 baseline floor
Full Gate 决定是否值得上正式算力 多 seed 稳定、配置与环境快照完整、失败可恢复
Promotion Gate 决定是否进入“值得继续研究”列表 holdout 表现、鲁棒性、隐私风险、下游任务都满足阈值

指标设计建议

不要只看视觉质量

医学影像生成里，“看起来像”往往不是最关键的。更重要的是解剖一致性、病灶保真、跨中心泛化和下游任务效用。

LLM reviewer 只能做辅助

它适合总结日志、梳理失败模式、生成 review 草稿；不适合单独裁定临床真实性或科研可发表性。

promotion_score = 0.30 * anatomical_fidelity + 0.25 * downstream_utility + 0.20 * cross_site_robustness + 0.15 * reproducibility + 0.10 * sample_quality - cost_penalty hard_constraints: - privacy_leakage == 0 - catastrophic_artifact_rate < threshold - all_required_logs_present == true - at_least_3_seeds_succeed == true

常见误区

误区 1：把 harness 等同于 prompt engineering

这会导致系统无法积累状态，也无法独立比较不同版本的运行质量。

误区 2：先做多 agent，再想评估

没有评估的多 agent 往往只会把复杂度和成本提前放大。

误区 3：只存总分，不存原始轨迹

这样外环根本没有足够证据做失败归因，Meta-Harness 会退化成盲目 hill climbing。

误区 4：让 LLM judge 替代硬规则

在高风险场景里，LLM judge 只能补充，不能取代 deterministic checks 和 domain-specific metrics。

误区 5：把所有规则都塞进单个大文件

规则必须结构化、可链接、可过期、可版本化。单个巨型手册只会快速腐烂。

最稳妥的落地方式不是一步到位做“全自动”，而是先做固定 harness，再做 trace registry，再做半自动外环，最后才做受控自动优化。
