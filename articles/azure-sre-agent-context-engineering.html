Azure SRE Agent 的经验：为什么 100+ 工具和 50+ 专家代理，最后不如 5 个宽工具

这篇是少见的生产经验复盘：微软在构建 Azure SRE Agent 时发现，复杂系统的很多问题并不是靠加更多专家代理和更多工具解决，反而常常要靠删减、收敛和外化状态来解决。

如果你喜欢看‘踩坑后真正改了什么’，这篇非常值得读。
它不是从零讲概念，而是站在一个要处理真实 SRE 场景的 agent 视角，回顾工具设计、代理编排、上下文管理和数值分析为什么会失灵。
文章最打动我的地方，是它敢承认：他们一开始走的很多路，看上去很合理，但在生产里并不好用。

一、从 100+ 工具到 5 个核心工具：复杂度有时来自帮助过度

文章最著名的结论之一，是团队最开始做了 100 多个工具、50 多个专门代理，后来发现这并没有带来预期中的可靠性。

工具太多导致模型在发现、选择和组合时负担过重；很多工具功能重叠，描述边界又不够清晰，反而让 agent 更容易犹豫、走偏或循环。

最终他们收敛到更少的宽工具，尤其是像 az 和 kubectl 这样的成熟 CLI。

这个决定很有启发性：模型往往更擅长驱动少量表达力强的通用接口，而不是在海量窄接口中做精细导航。

与其把世界切得过碎，不如给它几把好用的大锤，再配合外部状态和良好验证。

二、多代理交接为何失败：分工不是越细越好

很多人直觉上会觉得，多代理系统只要专业分工足够细，就能像组织一样高效协作。

微软的经验却显示，交接超过四次后，系统就很容易出现信息丢失、上下文漂移、重复探索甚至循环依赖。

我很赞同文章的这个判断。代理之间的 handoff 并不是免费的组织设计，它本质上是一种上下文压缩与重建。

每多一次 handoff，就多一次状态损失和语义变形的机会。

因此除非子问题真的高度独立，否则过度分工可能比少量 generalist 更糟。

三、把知识从 prompt 迁到文件，把输出从上下文迁到数据源

文章另一个非常成熟的点，是强调把知识与状态从 prompt 中迁出来。

他们逐渐把知识封装进文件、skill 或结构化工件，而不是继续往系统提示里堆细则。

同时，大量工具输出、指标结果和中间分析也不再直接塞回上下文，而是写入会话文件或由代码解释器进一步处理。

这一点几乎和 OpenAI、Anthropic 的经验形成了交叉印证。

一旦任务涉及大量指标、日志和数值计算，让模型直接在上下文里盯着原始数据分析，既贵又不稳。

更好的做法是让代码解释器做计算，让文件系统保存数据，让模型负责 orchestration 和判断。

四、code interpreter 的启示：LLM 更像指挥，不像计算器

文章把 code interpreter 的引入称作一个关键转折点，我非常同意。

在 SRE 场景里，指标聚合、时间序列比较、表格变换、分布统计，本来就不该靠模型在 token 里‘心算’。

当这些工作下沉到代码执行环境后，模型终于可以回到自己更擅长的位置：制定下一步调查策略、解释结果、选择工具链。

这个判断其实很普适。很多 agent 失败，并不是因为规划错了，而是因为系统把不该由模型直接承担的机械计算也强行塞给它。

一旦你把计算外包给解释器，把状态外包给文件，把调用外包给宽工具，模型就更容易表现出真正的推理价值。

五、我的读后感：这是 context engineering 的‘减法哲学’

我觉得微软这篇最宝贵的地方，在于它没有迷信“更多组件 = 更强系统”。

无论是专家代理、窄工具、长 prompt，还是过度嵌套的 orchestrator，最后都在提醒我们：系统复杂度本身就是可靠性的敌人。

这篇文章和 Anthropic 的 context engineering 文章放在一起看，几乎形成一条清晰主线：

能少给就少给，能外化就外化，能用宽工具就别拆碎，能让代码解释器算就别让模型算。

这不是保守，而是对模型认知特性的尊重。

对真正做生产 agent 的团队来说，这篇文章提供的不是口号，而是一种极其实用的纠偏视角：

当系统变得越来越复杂时，先试着删东西。

更多 bad example / good example

Bad example
为了覆盖面设计上百个窄工具。
Good example
收敛到少量宽工具，让模型复用已有 CLI 先验。
Bad example
把复杂任务拆给几十个专家代理层层接力。
Good example
优先用少量 generalist，仅在必要时才做明确的子任务分解。
Bad example
把知识和状态都堆进系统 prompt。
Good example
把知识迁到文件和 skills，把状态迁到外部工件。
Bad example
让模型直接在上下文里做复杂数值分析。
Good example
用 code interpreter 处理计算，模型负责解释和决策。
Bad example
系统一复杂就继续加组件。
Good example
优先做减法，检查是否存在工具冗余、代理冗余和上下文冗余。

重点结论

结论 1

少量宽工具往往比大量窄工具更适合 agent。

结论 2

多代理 handoff 有很高的语义损耗成本。

结论 3

LLM 更适合编排和解释，不适合承担大量机械计算。

结论 4

知识与状态应该尽量外化到文件、skills 和数据工件。

结论 5

生产可靠性常常来自减法，而不是加法。
