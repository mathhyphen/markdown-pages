Harness design for long-running application development：让 AI 在数小时任务中保持节奏的三代理架构

Anthropic Labs 团队的 Prithvi Rajasekaran 在两个领域（主观审美与客观正确性）各建一套 harness，最终都收敛到同一个 GAN 启发的三代理架构：Planner 规划、Generator 构建、Evaluator 评估。

这篇文章的核心洞察是：对于超越模型独立能力的复杂任务，将生成与评估分离，并让一个独立评估者（带 Playwright MCP）持续校验，是让整个系统在数小时运行中保持诚实的关键。
而当模型升级（Opus 4.5 → 4.6）时，harness 也需要重新审视——某些原本必要的组件（如 sprint 分解）可能变成多余。 Evaluator 的价值在于任务落在模型能力边界之外的那一段。 ——这是整篇文章最值得记住的一句话。

一、朴素实现的两类失败：状态丢失与自我美化

文章首先回顾了早期长任务 harness 的经验教训。在更早的实验里，团队用 initializer agent 将产品 spec 分解成任务列表，然后用 coding agent 逐个 feature 实现，并通过 context reset 在会话之间传递状态。

但有两个问题始终存在：

第一类是上下文焦虑（context anxiety）。模型在接近上下文窗口上限时，会产生"时间不够了"的紧迫感，从而提前收尾工作。这在 Claude Sonnet 4.5 上尤为明显。Context reset（清空上下文窗口并用结构化工件传递状态）可以解决这个问题，但增加了编排复杂度、token 开销和延迟。

第二类是自我评估失真。当要求 agent 评估自己产出时，它几乎总是自信地赞美，即使产出明显平庸。这一点在主观任务（如设计）上尤为严重；在有客观验证标准的任务上，agent 也常常对自己的错误判断过于宽容。

解决方案是将生成与评估分离：让另一个 agent（而非生成者自己）来评判工作质量。这看似简单，但调优一个独立的 evaluator 让它保持适度的怀疑态度，远比让生成者学会自我批评要容易得多。

二、GAN 启发的设计评估体系：把主观审美变成可评分项

作者首先在前端设计领域做实验，因为那里的自我评估失真最直观：没有干预时，Claude 默认生成安全、呆板、技术上能跑但视觉上毫无特色的布局。

两个关键洞察：

第一，审美虽然无法被完全量化，但可以通过评分标准（criteria）转译为模型可以评估的描述性判断。"这个设计美吗？"很难回答一致，但"这个设计是否符合我们的设计原则？"给 Claude 提供了具体的评判依据。

第二，将生成与评估分离创造了一个反馈循环，驱动生成器向更强输出迭代。

作者制定了四个评分标准，要求 Generator 和 Evaluator 在提示中都使用：

  • Design quality（设计质量）：设计是否像一个连贯整体而非零件堆砌？颜色、排版、布局、图像等细节是否共同营造了独特氛围？
  • Originality（原创性）：是否有定制化决策的证据，还是模板布局、库默认值和 AI 生成模式？未修改的库存组件或明显的 AI 风格（如白色卡片上的紫色渐变）在此失分。
  • Craft（工艺）：技术执行质量：排版层级、间距一致性、色彩和谐度、对比度。这是能力检查而非创意检查。
  • Functionality（功能性）：独立于美学的可用性：用户是否能理解界面作用、找到主要操作、在不猜测的情况下完成任务？

作者有意强调 Design quality 和 Originality，而非 Craft 和 Functionality——因为 Claude 在后两项上本来就能做好，但在前两项上常常输出平庸。评分标准明确惩罚了通用的"AI slop"模式，迫使模型在设计上承担更多风险。

Evaluator 使用 Playwright MCP 直接与运行中的页面交互，在评分前主动导航页面、截图并仔细研究实现细节，而非仅仅评估静态截图。每轮迭代通常耗时真实时钟时间，完整运行长达四小时。

作者通过 few-shot 示例校准 Evaluator，包含详细的评分分解，确保其判断与偏好一致，并减少跨迭代的评分漂移。

在一次实验中，要求模型为一家荷兰艺术博物馆创建网站。到第九次迭代，它产出的是一个干净、深色主题的登录页——视觉上精致，但大致在预期范围内。然后，第十次迭代，它彻底放弃原有方案，重新构想了网站形态：一个 CSS 透视渲染的 3D 房间，带有棋盘格地板，艺术品以自由形式挂在墙面，用门口代替滚动或点击来在画廊房间之间导航。这是作者从未在单次生成中见过的创意飞跃。

三、三代理架构全栈开发：Planner × Generator × Evaluator

带着这些设计实验的教训，作者将 GAN 启发模式应用于全栈开发——代码审查和 QA 在结构上与设计评估者扮演相同角色。

架构组成

  • Planner：早期的 long-running harness 要求用户预先提供详细 spec。作者希望将这个步骤自动化——Planner 接收一个 1-4 句的简单提示，扩展为完整产品 spec。它被要求在范围上保持雄心，专注于产品上下文和高层技术设计，而非详细技术实现（以避免 spec 错误级联到下游实现）。Planner 还被要求寻找将 AI 功能编织到产品 spec 中的机会。
  • Generator：采用早期 harness 中的单 feature 工作方式，按 spec 逐个 sprint 实现。每个 sprint 使用 React + Vite + FastAPI + SQLite（后改为 PostgreSQL）技术栈，并在 sprint 结束时自我评估，然后交接给 QA。它还使用 git 做版本控制。
  • Evaluator：使用 Playwright MCP 点击运行中的应用，测试 UI 功能、API 端点和数据库状态。它根据 sprint 中发现的 bug 以及一套标准（改编自设计实验，覆盖产品深度、功能性、视觉设计和代码质量）来评分。每个标准都有硬性阈值——任何一项低于阈值，sprint 即判定失败，Generator 收到详细反馈。

Sprint Contract 机制

每个 sprint 之前，Generator 和 Evaluator 协商一份 sprint contract：在任何代码编写之前，双方就"完成标准"达成一致。这之所以必要，是因为产品 spec 在高层面上是故意保持模糊的，需要一个步骤来弥合用户故事与可测试实现之间的差距。Generator 提出它将构建什么以及如何验证成功，Evaluator 审查提案以确保 Generator 在构建正确的内容。双方迭代直到达成一致。

Context Reset 的简化

在早期 harness 中，Context Reset 是关键解锁点，因为 Sonnet 4.5 表现出明显的"上下文焦虑"。但 Opus 4.5 大致消除了这种行为，因此在这个 harness 中，Context Reset 被完全放弃。代理作为跨越整个构建的单一连续会话运行，Claude Agent SDK 的自动压缩沿途处理上下文增长。

四、两次实验的成本与质量对比

第一版 harness 使用 Claude Opus 4.5，将用户提示分别针对完整 harness 和单代理系统运行比较：

实验一：2D 复古游戏制作器

提示："Create a 2D retro game maker with features including a level editor, sprite editor, entity behaviors, and a playable test mode."

Harness 类型时长成本
Solo（单代理）20 min$9
Full harness（三代理）6 hr$200

Harness 成本是单代理的 20 倍以上，但输出质量差异立即显现：

  • 单代理版本：界面布局浪费空间（固定高度面板让视口大部分为空）、工作流僵硬、游戏核心功能完全损坏（实体出现在屏幕上但没有任何输入响应）
  • 完整 harness 版本：Canvas 使用全视口、面板尺寸合理、界面具有符合 spec 设计方向的统一视觉身份。更重要的是，游戏的核心功能可用——物理效果有粗糙之处，但基本的东西确实能跑

Evaluator 在每个 sprint 都通过 Playwright 进行了详细 QA。Sprint 3 仅一个 sprint 就包含 27 项验收标准，涵盖层级编辑器、实体生成点删除、动画帧重排序 API 等。Evaluator 的发现具体到可以直接操作：例如，它发现 fillRectangle 函数存在但没有在 mouseUp 时正确触发；删除键处理程序的条件判断逻辑错误（应该是 `selection || (selectedEntityId && activeLayer === 'entity')`）。

实验二：DAW（数字音频工作站）

提示："Build a fully featured DAW in the browser using the Web Audio API."

阶段时长成本
Planner4.7 min$0.46
Build（Round 1）2 hr 7 min$71.08
QA（Round 1）8.8 min$3.24
Build（Round 2）1 hr 2 min$36.89
QA（Round 2）6.8 min$3.09
Build（Round 3）10.9 min$5.88
QA（Round 3）9.6 min$4.06
Total3h 50m$124.70

Generator 运行了两个多小时而没有早期 harness 所需的 sprint 分解。QA 在第一轮发现：应用看起来令人印象深刻，AI 集成也很好，但几个核心 DAW 功能是"仅展示"而没有交互深度（clip 不能拖动/移动、没有乐器 UI 面板、没有可视化效果编辑器）。这些不是边缘情况——它们是使 DAW 可用的核心交互，而 spec 明确要求它们。第二轮 QA 又发现了录音功能 stub、AI 生成关卡的可玩性有限等问题。

最终产出：一个功能完整的音乐制作程序，包含工作排列视图、混音器和传输控制，全部在浏览器中运行。Agent 能够通过提示自主驱动整个创作流程——设置 tempo 和 key、铺设旋律、建立鼓轨道、调整混音电平、添加混响。

五、Opus 4.6 时代的 harness 简化

第一个 harness 结果令人鼓舞，但它也很笨重、缓慢且昂贵。逻辑上下一步是找到简化方法而不降级性能。

一个核心原则：harness 中的每个组件都编码了一个关于"模型自身无法做到什么"的假设。随着模型改进，这些假设值得重新审视——它们可能不正确，也可能很快过时。

在尝试激进简化失败后（难以判断哪些组件是真正承重的），作者改为更系统的方法：每次移除一个组件，审查对最终结果的影响。

移除 Sprint 构造

Opus 4.6 发布了，它带来了实质性改进：更仔细的规划、更长时间维持 agentic 任务、在更大代码库中更可靠地操作、更好的代码审查和调试能力来捕捉自己的错误，以及显著改进的长上下文检索。所有这些都是 harness 原本被构建来补充的能力。

因此，作者移除了 sprint 构造。Planner 和 Evaluator 被保留（因为每个都继续添加明显价值），但 Evaluator 从每个 sprint 评分移动到运行结束时单次通过。

这改变了 Evaluator 的负载特性——其有用性取决于任务相对于模型可靠独立完成能力的边界。在 Opus 4.5 上，边界很近：构建位于生成器能良好完成的能力边缘，Evaluator 在整个构建中捕捉有意义的问题。在 Opus 4.6 上，模型原始能力提高，边界向外移动——曾经需要 Evaluator 检查才能连贯实现的任务现在通常在生成器自身能力范围内。对于边界内的任务，Evaluator 成为不必要的开销。但对于仍然处于生成器能力边缘的部分，Evaluator 继续提供真正的提升。

实际含义：Evaluator 不是固定的"是/否"决策。当任务处于当前模型可靠独立完成能力之外时，它才值得付出成本。

六、我的读后感

这篇文章是作者对自己数月工作的系统性复盘，展示了如何在两个非常不同的领域（主观审美和客观正确性）上最终收敛到同一个架构模式。这本身就很有价值。

最有启发性的几点：

第一，分离生成与评估是打破"自信自夸"循环的最有效手段。这不是因为分离本身有什么魔法，而是因为它让调优evaluator变得 tractable——你可以单独调整它，而不必同时改变生成者的行为。

第二，harness 是对模型当前能力边界的描述，而不是永久固定。随着 Opus 4.5 → 4.6，某些原本必要的组件变成了多余。这不是 harness 设计失败，而是持续迭代的正常部分。

第三，Evaluator 的价值由任务所在的位置决定——当任务超出模型可靠独立完成的能力边界时，Evaluator 提供真正的提升。这意味着为不同能力的模型可能需要不同的 harness 策略。

第四，GAN 隐喻不是噱头。在设计领域，作者观察到"最好的设计是博物馆质量"这样的措辞如何推动设计向特定视觉收敛——这表明与标准相关的提示可以直接塑造输出特征。这是 GAN 思想在非图像领域的成功应用。

专业读后感

读后感 1

分离生成与评估之所以有效，是因为它将"让evaluator变得怀疑"这件事变得可调优——你可以在不改变生成者行为的情况下单独优化它。

读后感 2

harness 是对模型能力边界的时刻快照，不是永久结构。随着模型改进，需要重新审视每个组件是否仍然承重。

读后感 3

Evaluator 不是"是/否"开关，而是一个连续的价值函数——当任务超出模型独立能力时，它提供真实提升。

读后感 4

"博物馆质量"这样的措辞能直接塑造输出特征——这意味着与评估标准一起传达的提示语言本身就是 harness 的一部分。

读后感 5

Planner 的价值在移除 sprint 构造后仍然存在——没有它，Generator 会低估范围，不先 spec 就开始构建。

重点结论

结论 1

分离生成与评估是打破 agent 自我美化评估循环的最有效手段。

结论 2

GAN 启发的评分标准（Design quality、Originality、Craft、Functionality）可以将主观审美转化为可迭代的 gradable 反馈。

结论 3

三代理架构（Planner、Generator、Evaluator）在主观（设计）和客观（全栈）两个领域都收敛到相同模式。

结论 4

Opus 4.6 改变了 harness 设计的前提——某些原本必要的组件（如 context reset、sprint 分解）可能变成多余。

结论 5

Evaluator 的价值是任务复杂度的函数：当任务超出模型可靠独立完成的能力边界时，Evaluator 提供真实提升。

结论 6

harness 是对模型能力边界的持续描述，需要随模型改进而重新评估每个组件是否仍然必要。
