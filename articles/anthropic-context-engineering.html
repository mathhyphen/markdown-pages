Effective context engineering for AI agents：上下文不是越多越好，而是越“刚好能决策”越好

这篇文章把 context engineering 讲得非常清楚：真正的问题从来不是“有没有上下文”，而是“给模型的这些 token 是否在当前时刻最值得占用注意力”。

如果说 harness engineering 是更大的系统观，那么这篇文章几乎就是 context engineering 的教科书。
它不断提醒读者：上下文窗口不是一个越大越保险的垃圾桶，而是一个昂贵、竞争激烈的注意力空间。
好的系统不是把一切都塞进去，而是让最相关、最可行动、最少歧义的信息在恰当时刻进入。

一、context engineering 与 prompt engineering 的真正区别

文章一开始就把概念分开：prompt engineering 更偏向那段直接对模型说的话，而 context engineering 则是持续策划“此刻要给模型看什么”。

这包括系统提示、工具描述、历史消息、检索到的文档、外部状态摘要、少量高价值示例等。

我很喜欢这种区分，因为它把问题从“写一段更聪明的话”升级成“设计一个信息供给系统”。

对于 agent 来说，真正的挑战往往不是单次提示词，而是随着任务推进，哪些信息该进、哪些该出、哪些该被压缩、哪些只该以链接或路径存在。

二、有限注意力预算：不是模型看不见，而是它没必要一直看

文章反复强调 finite attention budget。即便上下文窗口很长，注意力也不是免费的。

更多 token 会带来更高的计算代价，也会稀释模型对真正关键信号的聚焦。

因此“能塞进去”不代表“应该塞进去”。

这里我觉得 Anthropic 最值得借鉴的一点，是它要求我们把 token 当作预算来花。

你应该优先保留那些能改变当前决策的事实、约束和高质量示例；至于大段背景、冗余历史、重复说明、庞大原始数据，

很多时候更适合以路径、查询、摘要或按需检索的形式存在。

三、工具、示例与检索：好上下文的标准是高信号、低歧义、可按需展开

文章对工具设计的建议非常实战：工具要少而清楚，描述要自包含，避免让模型面对过多功能重叠的接口。

如果工具之间边界模糊，模型不仅难选，还会在上下文里浪费注意力比较它们的差异。

在 few-shot 示例上，文章也反对“样例越多越保险”。更好的做法是挑少量 canonical、互补且能代表策略的例子。

一堆近似样例往往只会放大表层模式，反而遮蔽真正的决策原则。

至于检索，Anthropic 倾向于 just-in-time retrieval 和 progressive disclosure：

不是把整个知识库搬进 prompt，而是给模型一种在需要时取用正确材料的机制，比如文件路径、结构化索引、存好的查询或链接。

四、长任务中的 context rot：上下文也会腐烂

这篇文章里一个非常重要的概念是 context rot。随着任务延长，历史消息里会积累过时计划、已解决问题、错误分支和重复指令。

如果系统不清理，这些内容就会像旧地图一样继续干扰当前决策。

Anthropic 的应对方式包括 notes 文件、阶段性总结、compaction 和 sub-agents。

我尤其认同 notes 的作用：它是一种低成本、可人工检查的外部记忆，把真正重要的发现从庞杂对话中拎出来。

这样下一轮不必再从所有历史里打捞线索，只需要读 notes 就能重新上手。

从工程角度看，context rot 是比“上下文不够长”更常见也更隐蔽的问题。

很多人直觉上以为丢信息才危险，但在实际 agent 系统里，带着太多脏信息继续工作往往更危险。

五、我的总评：这篇文章在教你如何尊重模型的工作记忆

我会把这篇看作所有 agent 平台工程师都应该反复回读的基础文档。

它不是在贩卖某个技巧，而是在训练一种信息卫生观：上下文是一种高价值、低容量、需要持续清扫和编排的资源。

它也解释了为什么一些团队在有了更大窗口后，系统表现并没有同步变好。

因为如果你不改变供给方式，只会更容易把更多噪声塞进去。模型不是因为‘没看到’而犯错，而是因为它被无关历史稀释了判断。

专业上我最认可它的一点，是把 notes、检索、compaction、sub-agent 都放进同一框架里理解。

这些不是孤立技巧，它们共同服务于一件事：让模型在每个时刻只为真正该做的决策付出注意力。

更多 bad example / good example

Bad example
把能想到的所有背景都塞进 prompt。
Good example
只保留当前决策需要的信息，其余内容按需检索或外化存储。
Bad example
给模型十几个相似工具，靠它自己猜差别。
Good example
压缩工具面，让每个工具边界清晰、描述自包含。
Bad example
用一堆相似 few-shot 堆满上下文。
Good example
只选少量 canonical 且互补的示例，传达原则而不是表面样式。
Bad example
把旧计划和旧错误分支一直留在历史里。
Good example
通过 notes 和 compaction 定期清扫 context rot。
Bad example
一旦模型出错，就继续加更多说明。
Good example
先检查是不是给了过多噪声，削减无关信息往往更有效。
