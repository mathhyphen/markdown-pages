Claude Code 15 条高频实战技巧：把 Boris 的工作流整理成一页工具手册

这篇内容来自 Boris Cherny 在 2026 年 3 月 30 日公开分享的一组 Claude Code 隐藏与高频功能，AlchainHust 的转发长文把它重新带回中文圈。我这里不做逐条搬运，而是把原始技巧整理成更适合开发者直接吸收的工具手册：哪些是跨端接力，哪些是自动化，哪些是并行工作流，哪些是在给 agent 真正的“眼睛”和“手”。

如果你还在让一个 Claude 会话独自背着所有任务、所有上下文、所有批准请求一路硬扛，那问题往往不是模型不够强，而是你的工作流对它有点过于“单身主义”。

一、为什么这组技巧值得单独放进工具区

这批内容最有价值的地方，不是“Claude Code 又多了 15 个命令”，而是它把 Claude Code 从一个终端聊天器，重新呈现成一个可跨设备、可自动化、可并行、可封装能力的运行时环境。

很多开发者用 agent 时，习惯仍然停留在“开一个会话，写一个 prompt，盯着它跑”。而 Boris 这组技巧展示的是另一套范式：移动端接力、云端和本地切换、hooks 注入、批量 worktree 扇出、前端自动验证、定时 loop、定制 agent。换句话说，它不是在教你“怎么把一句提示词说得更圆”，而是在教你怎么把 agent 变成一个真正有操作系统味道的工作台。

所以它适合放在工具板块，而不是论文板块。这里讲的不是一个新 benchmark，也不是一个研究结论，而是开发流程里立刻能拿来改变产能和节奏的具体做法。

二、15 条技巧整理版

下面这张表保留了原始内容的主干，同时加了一列“为什么重要”，方便你快速判断哪些值得先上手。

# 技巧 核心动作 为什么重要
1 移动端 Claude Code 在 iOS / Android 上进入 Code 视图，直接看改动、继续任务、处理批准。 把“必须坐到电脑前才能继续 agent”这个前提打掉，碎片时间终于不只能刷消息。
2 /teleport 与 /remote-control 把云端会话拉回本地，或从手机 / Web 控制本地运行中的 Claude 会话。 会话不再被设备绑死，连续工作比“在哪台机器上开了它”更重要。
3 /loop 与 /schedule 把代码审查跟进、PR 清理、反馈轮询这类重复任务定期自动执行。 重复性的维护工作可以逐步从“手动提醒自己”迁移到“让 agent 定时处理”。
4 Hooks 在 SessionStart、PreToolUse、PermissionRequest、Stop 等生命周期节点注入确定性逻辑。 这相当于给 agent 增加硬边界和固定流程，少靠情绪，多靠机制。
5 Dispatch 把桌面端 Claude 作为远程协作执行器，在不坐电脑前时处理 Slack、邮件、文件等事务。 它把“我离开电脑了”从流程中移除，agent 真正开始替你守着工作台。
6 给前端任务浏览器验证能力 用 Chrome 扩展或类似能力，让 Claude 自己看到页面结果并反复迭代。 前端质量的分水岭往往不是 prompt，而是模型有没有机会验证自己写出来的东西。
7 桌面端自动拉起并测试 Web 服务 让 Claude 自动启动本地 Web 服务，并在内置浏览器里检查运行效果。 这会把“改代码”和“跑起来看”连成闭环，减少你频繁手动来回切换。
8 Fork 会话 用 /branch 或 claude --resume <session-id> --fork-session 从现有会话分叉。 你可以保留主线，同时为另一个假设或修复路线开一条支线，避免把一个上下文揉成面团。
9 /btw 侧问 在 agent 忙主任务时问一个旁支问题，而不打断当前执行。 这很小，但很像真正的多任务体验，尤其适合边跑边查边确认。
10 Git Worktrees 用 claude -w 或桌面端 worktree 入口，为同仓库并行任务创建独立工作副本。 真正的并行不是把十件事塞进同一个会话，而是给十件事各自的空间。
11 /batch 扇出大规模修改 让 Claude 采访你需求后，把任务分发给大量 worktree agents 并行完成。 对于批量迁移、机械性重构这类任务，并行扇出带来的收益往往远大于继续手工拆任务。
12 --bare 在 SDK 或非交互场景里显式关闭默认加载，减少启动时搜索本地配置的开销。 它提醒我们：交互式体验友好的默认值，不一定适合程序化调用。
13 --add-dir / /add-dir 把其他仓库或目录加入当前 Claude 会话的可见与可操作范围。 跨仓库开发时，上下文与权限常常一起卡脖子，这个入口正好同时解决两件事。
14 --agent 与自定义 agents 在 .claude/agents/ 里定义有专属系统提示、工具权限、模型配置的 agent。 把固定角色做成 agent，比每次重写一段长 prompt 更稳定，也更像工程资产。
15 /voice 在 CLI、桌面端或手机侧用语音输入继续驱动 Claude。 输入方式一旦放松，很多原本懒得写下来的意图和修正，就会更自然地进入工作流。

三、这些技巧其实在指向什么工作流

1. 把会话从“设备对象”变成“可迁移状态”

移动端、/teleport、/remote-control、Dispatch、本质上都在干一件事：让 session 跟着任务走，而不是跟着设备走。只要这一点成立，agent 才真的能进入日常工作节奏。

2. 把重复维护工作编译成自动化

/loop、/schedule、hooks 这一组很关键。它们让 Claude Code 不只处理“此刻这次任务”，还开始接管那些原本散落在你脑内待办里的例行琐事。

3. 给前端与交互任务真正的反馈回路

前端问题最怕闭门写代码。浏览器验证、自动起服与内置测试，把“生成”与“检查”绑在一起。这类能力的价值，通常比再把 prompt 润色三轮更大。

4. 用 worktree 而不是上下文污染来做并行

Fork 会话、Git worktrees、/batch 都指向同一个思想：并行最好的实现方式，不是把更多信息塞进一个对话框，而是把任务拆给多个有边界的执行单元。

5. 把“角色”和“访问范围”做成显式配置

--add-dir 和 --agent 说明 Claude Code 的工程化不是抽象口号。目录边界、工具边界、角色边界，都可以被做成明确的运行时配置，而不是靠人每轮口头说明。

四、最值得拿走的几个 idea

Idea 1：Claude Code 更像一个 agent runtime，而不是高级聊天框

这组技巧最核心的启发，是把 Claude Code 当成一个能调度会话、封装角色、连接设备、运行 hooks、管理目录权限的工作环境来看。视角一变，很多功能 suddenly make sense。

Idea 2：验证能力比“提示词雕花”更值钱

Boris 对前端工作反复强调的一点很对：如果模型能看到结果，它就更有机会自己修正。很多 agent 工作流瓶颈不在生成，而在缺乏反馈。

Idea 3：并行不是奢侈品，而是默认生产方式

当任务可拆、仓库支持 worktrees、会话能分叉时，单线推进反而会变成瓶颈。真正成熟的 agent 工作流，天然会往扇出、分支、汇总收敛。

Idea 4：把重复角色做成资产，比重复解释更省心

自定义 agent、本地 hooks、额外目录授权，这些都在鼓励一种更工程化的习惯：把长期稳定的规则沉淀为配置，而不是每次重新对模型“晓之以理”。

Idea 5：移动端不是玩具，而是打断恢复器

很多人低估手机入口的价值。它未必适合重度实现，但非常适合确认、批准、续跑和转移控制权。对长任务来说，这能显著降低“人离开电脑就停机”的损耗。

五、如果今天就想开始用

不需要一次吃完 15 条。比较稳妥的起步顺序是下面这五步。

第一步：先把反馈回路补上

如果你做前端或带界面的工作，优先解决“Claude 能不能自己看结果”。这是收益最大的一步。

第二步：把一个重复任务改写成 loop

比如定时扫 PR、定时处理 review comments、定时整理反馈。只要成功一个，你就会知道自动化不只是演示用。

第三步：开始用 worktree 跑并行

挑两个互不冲突的小任务，别再塞进同一会话里。用隔离换来清爽，比用记忆力硬扛轻松很多。

第四步：把一个常见角色做成自定义 agent

例如只读审查员、文档整理员、前端验收员。把角色从 prompt 升级成配置，后面会省掉大量重复解释。

第五步：再考虑移动端接力与语音输入

这些能力不会直接让代码质量暴涨，但会明显改善任务连续性。很多时候，真正贵的不是模型费用，而是人类频繁掉线。
