AutoHarness：让模型自己合成 code harness，为什么这是 Harness Engineering 的下一步

这篇 DeepMind 新论文最有意思的地方，不是“模型又会写代码了”，而是它把 harness 从人工手写的外围胶水，推进到由模型自动合成、自动迭代、自动收敛的可执行控制层。对 Harness Engineering 这个系列来说，这几乎就是自然续篇。

一句话概括：以前我们讨论的是“怎么给 agent 搭 harness”；AutoHarness 讨论的是“能不能让 agent 自己把 harness 写出来，而且写得比直接裸跑更稳、更便宜、甚至更强”。

一、为什么它应该放进 Harness Engineering

如果只看标题，这篇论文像一篇“代码生成能力更强了”的 paper；但如果看它真正处理的问题，你会发现它更像一篇典型的 harness 论文。

作者关心的不是语言模型会不会下一个“聪明”的动作，而是模型在外部环境里会不会频繁做出根本不合法的动作。论文举的例子非常直接：在 Kaggle GameArena 的国际象棋比赛里，Gemini-2.5-Flash 的失败中，有 78% 归因于非法走子，而不是战略判断失误。

这正是 Harness Engineering 最典型的问题设定。也就是说，系统真正缺的不是更多 token、更多思维链，而是一个把模型输出变成“环境可接受动作”的执行中介层。

所以它不应该被放进纯 Papers 区当成一篇泛 AI 论文阅读，而应该进入 Harness Engineering：因为它讨论的是 agent 的运行边界、动作合法性、外部验证和控制回路，这些都属于 harness 的地盘。

二、论文到底做了什么

论文来自 Google DeepMind，题目是 AutoHarness: improving LLM agents by automatically synthesizing a code harness。作者把 harness 定义成连接 LLM 与任务环境的“glue / plumbing”。关键推进在于：他们不再手工写这层 glue，而是让 Gemini-2.5-Flash 在环境反馈下，自动合成并迭代修正这层代码。

实验场景主要是 TextArena 的 145 个文本游戏环境。论文报告称，自动合成出来的 harness 在这些 1-player 和 2-player 游戏中消除了非法动作，并让较小的 Gemini-2.5-Flash 反而超过了更大的 Gemini-2.5-Pro。

更激进的一步是，他们把方法继续推到“harness-as-policy”：不只是写一个验证器，而是直接让模型把整套策略编译成代码。这样测试时甚至不必再调用 LLM。论文在 16 个 TextArena 单人游戏上报告，最终的 code-policy 平均 reward 达到 0.870，高于 GPT-5.2-High 的 0.844，也高于 Gemini-2.5-Pro 的 0.707。

这篇论文真正惊艳的地方，不在于某个单点成绩，而在于它证明了一个方向：很多 agent 问题其实不是靠更强在线推理来解决，而是靠离线合成一个更好的执行壳层。

三、方法核心：code-as-harness

这篇论文的方法可以概括成三个层次。

第一层是 harness-as-action-verifier。先让 LLM 提议动作，再由代码函数判断它是否合法；如果非法，就带着环境反馈继续 refine 代码或动作生成逻辑。这是一种“可学习的 rejection sampler”。

第二层是 harness-as-action-filter。不是只检查单个动作，而是先生成候选动作集合，再过滤出合法动作，把“合法性”从 prompt 里的软约束，搬到代码里的硬约束。

第三层是 harness-as-policy。代码不再只是守门，而是直接产出动作。运行时不再依赖模型做最终决策，推理成本几乎被编译掉了。

更重要的是，作者没有把生成 harness 当成一次性的 codegen，而是把它定义成程序搜索问题。它们维护多个代码候选，用 Thompson sampling 在候选树里选下一个 refinement 节点，LLM 充当 mutation operator，环境则充当 critic。这个结构很像“程序合成版的 agent loop”。

四、这个 idea 的优点

优点 1：把软提示升级成硬约束

很多 agent 失败，并不是因为模型完全不懂规则，而是“懂，但执行时还是会漏”。AutoHarness 的核心价值，是把“请遵守规则”从 prompt 里的礼貌提醒，变成代码里的可执行约束。这种约束是外部可验证的，比链式思维更稳。

优点 2：把环境反馈纳入训练闭环

传统 prompt 调整往往靠人类主观观察；AutoHarness 则直接用环境反馈当 critic。动作是否合法、reward 如何、哪一步该 refine 哪段函数，全部由运行结果驱动。这种闭环很符合 Harness Engineering 的工程精神：少猜，多验证。

优点 3：让小模型借助外壳超过大模型

这篇论文最有启发的地方，是“小模型 + 专用 harness”可能优于“大模型裸跑”。这说明很多我们以为必须靠更贵模型解决的问题，实际上可以通过更好的 runtime 设计来解决。

优点 4：把一次性推理转成可复用资产

普通 agent 调一次模型，价值会随上下文结束而消失；而合成出来的 harness 是代码资产，可以保存、复用、版本化、回归测试、甚至形成库。论文结尾也明确提出未来可以构建 reusable harness library。这一点和你这个项目当前的 Harness 主题非常契合。

优点 5：有机会把在线成本编译掉

当 harness 进一步变成 policy，运行时几乎不再需要模型调用。论文给出的结论非常实用：性能更高、测试成本几乎为零，而对比的 GPT-5.2 / GPT-5.2-High 实验成本大约是 640 美元。对真实工程团队来说，这个方向非常诱人。

优点 6：把 Harness Engineering 推向自动化

以前我们默认 harness 是平台工程师手工写的；这篇论文等于在说，未来平台工程师更像 harness architect，而不是每个环境都手写 verifier 的人。也就是说，工程师开始设计 harness synthesis loop，而不是单个 harness 本身。

五、需要冷静看的边界

这篇论文很强，但也有几个边界需要讲清楚。

第一，它当前验证最充分的环境是 TextArena 文本游戏。游戏环境通常规则清晰、反馈明确、非法动作容易判断，这比现实世界应用友好得多。真实业务系统往往充满隐式约束、权限副作用、跨服务状态和延迟反馈。

第二，这个方法依赖环境能给出足够明确的 critic 信号。若环境本身没有可靠的合法性判断，或者 reward 太稀疏、太晚，搜索和 refinement 的效率可能明显下降。

第三，论文也承认 harness-as-policy 在 2-player 场景更难，因为它不仅要学合法性，还要学对手建模和搜索策略。这提醒我们：自动 harness 很适合“规则约束强、验证明确”的问题，但不自动等于“直接解决所有智能问题”。

所以我的判断是：它不是对 Harness Engineering 的替代，而是对其中“约束层自动生成”这一支线的重大加强。

六、在本站中应该怎么放

我建议把它放进 Harness Engineering 系列，而不是 Papers 区。更具体一点，我会把它作为该系列的新增末尾条目，像一个“下一阶段研究笔记”。

原因有三点。第一，它直接继承了 OpenAI 那篇 Harness Engineering 的核心主题：系统能力来自 harness，而不只是模型本身。第二，它与 Context Engineering 互补，前者管“该给模型什么信息”，后者管“即使给错了，外壳如何兜底”。第三，它和 Harness Design 构成很自然的延伸：从人工设计代理结构，走向自动合成控制壳层。

如果要给它一句系列内定位，我会写成：“Harness Engineering 的自动化版本：让模型自己写出约束自己的 harness。”

重点结论

结论 1

AutoHarness 不是单纯提升推理，而是在提升 agent 的执行外壳。

结论 2

它最有价值的思想，是把“规则遵守”从 prompt 软约束转为代码硬约束。

结论 3

它证明了小模型配专用 harness，可以超过大模型裸跑。

结论 4

它非常适合被纳入 Harness Engineering，而不适合被降格成普通论文阅读条目。

结论 5

对工程团队真正重要的启发是：未来我们也许不再只写 harness，而是写“生成 harness 的系统”。
