Harness engineering：在 agent-first 世界里如何把 Codex 变成工程生产力

这篇文章不是在讲“怎么把提示词写得更像咒语”，而是在讲：当代码主要由 agent 产出时，工程团队真正要建设的是环境、知识、边界、反馈回路与垃圾回收机制。

这是我最推荐先读的一篇，因为它给出了 harness engineering 的“组织级定义”。
OpenAI 团队用一个极端实验逼自己转型：新产品从空仓库起步，代码、测试、CI、文档、内部工具几乎都由 Codex 生成，
人类不直接下场写代码，而是把注意力转向“如何让 agent 持续可靠地产出”。文章真正想传达的不是炫技，而是工程角色的重写。

一、文章到底在回答什么问题

文章的起点很激进：团队在五个月里构建并交付了一个内部 beta 产品，而且坚持“0 行人工手写代码”的约束。

这样做的目的不是证明模型多聪明，而是强迫团队把一切妨碍 agent 交付的因素暴露出来。

所以作者真正要回答的问题是：当“写代码”不再是工程师的主要工作，工程学的核心到底迁移到了哪里？

文章给出的答案是：迁移到了 harness 上。这里的 harness 不是单一脚本，而是一整套“让 agent 能工作、敢工作、工作后还能被维护”的系统。

它包含仓库结构、知识组织、文档制度、测试工具、可观测性、lint 规则、质量评分、自动化 review 与持续清理流程。

如果这些东西缺位，agent 并不是简单地“输出差一点”，而是会以机器速度把局部坏模式扩散成系统性熵增。

二、逐节解读：从空仓库到工程角色重写

文章开头先用几个数字建立可信度：空仓库起步、约百万行代码、约 1500 个 PR、小团队却维持极高吞吐。

这部分的意义不在于“数字厉害”，而在于说明一个判断：只要上下文、边界和反馈回路搭好，agent 的交付速度会高到反过来改变组织制度。

换言之，传统针对“人类代码吞吐”的流程，未必还能直接套在 agent-first 团队上。

接着作者解释工程师的新角色。以前遇到任务失败，人类会倾向于自己补上那段代码；在这个实验里他们不能这样做，

所以每次失败都被迫回到更本质的问题：到底缺的是工具、知识、边界、计划模板，还是验证机制？

这种“永远不手写，而是补系统”的姿态，正是 harness engineering 的方法论核心。

三、让应用与仓库对 agent 可读：可读性从“给人看”升级为“给 agent 看”

文中很重要的一层是“application legibility”和“agent legibility”。前者指让应用本身对 agent 可操作、可验证：

比如每个 worktree 都能独立启动应用，把 Chrome DevTools 接进运行时，让 Codex 能看 DOM、截图、导航、复现 bug；

还把日志、指标、trace 暴露成 agent 可查询的观测面。这样，agent 不只是“改代码”，而是能自己验证修复是否真的成立。

后者则更深：仓库里的知识必须以 agent 能发现、能理解、能机械校验的形式存在。

作者反对把一切塞进一个巨大的 AGENTS.md，因为那会挤占上下文、让重点失焦、并且极易腐烂。

他们把 AGENTS.md 降级成“目录页”，真正的知识以 docs/、PLANS、QUALITY_SCORE、RELIABILITY 等结构化文档存在，再用 lint 与 CI 维持新鲜度。

这一段最有启发性的一句话可以概括成：凡是不在仓库里、且 agent 在运行时拿不到的知识，对 agent 来说就等于不存在。

这其实把很多团队的隐性流程暴露出来了——架构共识还躺在 Slack、Google Docs、口头习惯与资深工程师脑子里时，

你以为你在训练 agent，实际上你在让它持续缺文档工作。

四、架构、品味与吞吐：为什么 strict boundaries 比“精细规定写法”更重要

文章并不主张把 agent 管到每一行，而是强调“把不变量机械化”。具体做法包括：

固定业务域分层、限制依赖方向、在边界处解析和校验数据、用自定义 lint 强制结构化日志与命名、让错误信息本身就能给 agent 提示修复方向。

这是一种很强的平台工程思维：中心化地定义边界，局部放开实现自由。

我认为这一段最值得反复咀嚼。很多团队直觉上会把“AI 工程纪律”理解成更长的 prompt、更细的 code style 指令，

但作者的做法恰好相反：对具体写法放松，对架构边界收紧。原因很简单，边界是可机械执行的，审美争论却会迅速消耗上下文与评审注意力。

只要代码正确、可维护、对下一次 agent 运行依然清晰，它就达标；它不需要完美复刻某位资深工程师的行文腔调。

同时，文章指出高吞吐会倒逼 merge philosophy 改变。PR 周期更短、阻塞门槛更少、flake 往往通过后续运行修复而不是长时间卡住。

这听起来像“降低标准”，但作者的真实意思是：当 agent 产能远高于人类注意力时，等待的机会成本会陡增，

所以组织需要把“修正便宜、等待昂贵”的现实吸纳进流程设计。

五、自治不是终点：熵、垃圾回收与黄金原则才是长线战场

文章最后几节是全篇最成熟的部分。作者坦率承认：完全自治会复制仓库中已经存在的模式，好的坏的都会复制。

所以 agent 越能干，越会把局部不良模式迅速放大。团队一开始靠人类每周五花 20% 的时间清理“AI slop”，很快发现完全不可扩展。

于是他们把“人类品味”编码成 golden principles：优先共享 utility，避免手搓重复 helper；

不做 YOLO 式数据探测，而是在边界验证结构或使用类型化 SDK；定期运行后台 Codex 任务扫描偏差、更新质量等级、发起定向重构 PR。

这个机制被作者比喻成垃圾回收。它不是年度治理工程，而是持续的小额偿债。

这一段让我最有共鸣的地方在于：作者没有把 agent 看作终结技术债的机器，而是把它看作既会制造债、也能自动还债的系统成员。

真正的关键不是“有没有债”，而是你是否把债务识别、分级、清理和回归验证，变成了日常背景任务。

这也是 harness engineering 与传统 prompt engineering 最大的分野：前者关注的是系统生命周期，而不是单轮输出的漂亮程度。

专业读后感

读后感 1

这篇文章最重要的贡献，是把工程师的杠杆点从“写得更快”改写成“让 agent 更容易做对”。

读后感 2

它把仓库文档、架构约束、lint、可观测性、执行计划和重构流程都视作同一套控制系统的一部分，这个视角比单独谈 prompt 高一个层级。

读后感 3

文章也很诚实：它承认这套方法依赖大量前置投资，且未必能直接迁移到所有团队；尤其长期数年的架构一致性仍是未知数。

读后感 4

如果说传统工程文化强调 craftsmanship，这篇文章强调的则是 compounding taste：把品味编码一次，然后让系统持续执行。

更多 bad example / good example

Bad example
把仓库知识塞进一个巨大的 AGENTS.md，指望 agent 全记住。
Good example
让 AGENTS.md 只做目录，把设计、产品、质量、可靠性与执行计划拆进结构化文档，并用 lint/CI 检查新鲜度。
Bad example
应用界面对 agent 不可见，只让它看代码，最后靠人类人工 QA 收尾。
Good example
把 UI、日志、指标、trace 都纳入 agent 的可验证回路，让它能自己复现、修复、回归。
Bad example
对每一处实现细节过度规定，结果 prompt 冗长、规则互相冲突。
Good example
只把边界、不变量与架构规则机械化，局部实现允许 agent 自主展开。
Bad example
发现 AI 残渣后靠人类定期大扫除。
Good example
把黄金原则写进仓库，持续跑后台 cleanup/refactor 任务，像垃圾回收一样小步快跑。
Bad example
看到坏模式后只在 code review 里口头抱怨。
Good example
把抱怨升级成规则：文档更新、lint、结构测试、质量分和定向重构 PR。

重点结论

结论 1

先把仓库变成系统的知识基座，再追求更高自治。

结论 2

把应用验证面暴露给 agent，比继续堆 prompt 更能提升可靠性。

结论 3

边界和不变量要严，具体写法可以松。

结论 4

技术债处理要日常化、自动化，而不是项目化。

结论 5

harness engineering 的终极目标不是让 agent “会写”，而是让它“长期写得稳”。
