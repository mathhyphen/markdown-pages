Unlocking the Codex harness：App Server 如何把同一个 agent 核心接到多种客户端

这篇文章把“harness”从组织方法拉回到系统实现层：同一个 Codex 核心如何通过 App Server 暴露给 Web、CLI、IDE、桌面客户端与第三方集成。

如果上一篇告诉你 harness engineering 在团队层面是什么，这一篇就是系统层面的“拆机图”。
它解释了 Codex 为什么需要一个 App Server，以及为什么 agent 交互不适合被压扁成普通 API 的 request/response 形式。
从这篇开始，你会真正理解“harness”不仅是提示与工具，更是一套会话、事件和权限流转协议。

一、为什么必须有 App Server：同一个 agent 核心，需要跑在很多“壳”里

文章开场先把问题摆正：Codex 同时存在于 Web、CLI、IDE 扩展、macOS App 等不同表面。

如果每个客户端都各自重写一套 agent loop、状态管理和工具调用逻辑，系统很快就会分叉，功能和行为也会不一致。

所以 OpenAI 选择把“真正的 Codex”收拢到同一套 harness，再通过 App Server 统一暴露给各种前端。

这里可以把 App Server 理解成“agent 内核的外设总线”。

客户端负责 UI、交互和宿主环境适配；App Server 负责把线程、回合、工具、认证、配置与状态流稳定地暴露出来。

这让新表面能重用已有能力，也让协议升级可以集中进行，而不是散落在多个产品代码库里。

二、从 MCP 到 JSON-RPC：不是标准不重要，而是语义要够丰富

文章回顾了 App Server 的来源：最早 Codex CLI 是 TUI，后来做 VS Code 扩展时需要复用同一 harness。

他们一开始试过把 Codex 暴露成 MCP server，但在 IDE 场景下，很多交互并不适合被压缩成单次函数调用语义。

比如流式进度、差异展示、审批暂停、恢复连接，这些都更接近“长会话事件流”，不是简单工具调用。

因此 OpenAI 最终选了一个面向客户端的双向 JSON-RPC 变体。文章的重点并不是“JSON-RPC 比 MCP 更高级”，

而是强调：如果你的客户端想完整承载 agent 会话，协议必须能表达 richer session semantics。

标准是否通用是一回事，能不能忠实表示 agent 的生命周期是另一回事。

三、harness 的内部构成：agent loop 只是核心，不是全部

文中把 Codex harness 内部拆成三类关键能力。第一类是 thread lifecycle 和 persistence：

线程可以创建、恢复、fork、归档，并保留事件历史，这样客户端断开后还能回来重建时间线。

第二类是 config 与 auth，例如登录 ChatGPT、读取配置、管理默认值。第三类是 tool execution 和 extensions，

也就是在一致的策略模型下执行 shell/file 工具、接入 MCP 与 skills。

这一点很重要，因为很多人把“agent harness”理解成一段 while-loop：模型输出工具调用，系统执行，再把结果喂回去。

这只是最内核的一层。真正产品化的 harness 还必须处理线程持久化、身份认证、配置演化、审批、协议兼容性和 UI 事件映射。

一旦这些能力缺失，agent 也许能 demo，但很难成为稳定产品。

四、conversation primitives：为什么要有 item、turn、thread 三层语义

文章最值得学习的设计，是把 agent 交互拆成 item、turn、thread 三层。

item 是最小原子单元，可以是用户消息、工具执行、审批请求、diff、assistant message 等，并且有 started / delta / completed 生命周期。

turn 表示由一次用户输入触发的一整轮 agent 工作；thread 则是跨多个 turn 的持久会话容器。

这个分层解决了两个老问题。第一，UI 可以更忠实地表现 agent 工作过程：什么时候开始，什么时候流式增量，什么时候真正完成。

第二，系统可以在不同客户端之间迁移和恢复状态，因为 thread 是稳定持久层，turn 是一次任务单元，item 是渲染颗粒度。

对前端工程师来说，这套语义比“就给我一段最终文本”更麻烦；但对真实 agent 产品来说，它几乎是必需品。

配合 initialize 握手和 approval 流，这套协议还能让客户端在执行高风险动作前介入。

换句话说，App Server 不是只负责把消息搬来搬去，它还提供了人类监督和权限控制的天然插口。

五、客户端集成与协议选择：这篇文章的真正结论

在集成部分，文章展示了三类客户端形态。桌面与 IDE 场景通常把 App Server 二进制当成长驻子进程来跑；

Web 场景则在容器里启动 App Server，再由后端把事件通过 HTTP/SSE 往浏览器推。

这一设计的关键在于：浏览器标签页是易失的，真正的会话状态不能托付给前端。

文章随后比较了几种集成方式：MCP 适合把 Codex 作为一个可调用工具嵌入现有 MCP 工作流；

cross-provider harness 协议适合多供应商抽象，但容易只保留“共同最小能力”；

App Server 则适合你真的想要完整 Codex harness，包括事件流、模型发现、配置、认证和审批。

另外还有更轻的 Codex Exec 与 SDK，分别偏单次自动化与程序内嵌控制。

我的读法是：这篇文章在替 agent 产品设计者划线。

如果你只是想“调用一个模型帮我写点代码”，轻量协议就够了；如果你要做的是有持续状态、有审批、有断点续跑、有丰富 UI 的 agent 产品，

那么你最终一定会发明自己的 item/turn/thread 与状态恢复机制。OpenAI 只是先把这条路走了一遍。

更多 bad example / good example

Bad example
每个客户端各自重写 agent loop，导致行为分叉、升级困难。
Good example
把核心 agent runtime 收敛到同一 harness，再通过 App Server 对外暴露。
Bad example
把 agent 交互强行压成单次 request/response。
Good example
显式建模 item、turn、thread，以及 started / delta / completed 生命周期。
Bad example
把浏览器标签页当成长任务的状态源。
Good example
把线程与进度保存在服务端，让 Web 前端只做事件展示与控制入口。
Bad example
为了追求通用性，只保留最小公共子集能力。
Good example
在通用协议之外，为 rich agent 交互保留更完整的产品级语义。
Bad example
审批只是 UI 提示，不进入协议模型。
Good example
把审批请求做成协议内的一等事件，让 agent 在等待人类决策时显式暂停。

重点结论

结论 1

agent harness 的外部接口，最好是事件流，而不是纯函数调用。

结论 2

持久线程是长会话 agent 的基础设施，不是附加功能。

结论 3

越 rich 的客户端体验，越需要清晰的协议原语。

结论 4

Web 客户端应尽量无状态，任务状态留在服务端。

结论 5

如果你在做真正的 agent 产品，协议设计与模型能力同样关键。
